# X投稿の定期収集方法 調査メモ

Twitter API は制限が厳しいため使わない前提で、特定ユーザーの投稿を定期取得する方法を調査した。

## 結論

2025-2026年時点で、完全無料かつ安定して動く方法は存在しない。Twitter/X がサードパーティアクセスを厳しく制限しているため。

---

## 方法一覧

### 1. RSS ブリッジ系（Nitter, RSSHub 等）

- Nitter: 2023年後半以降、X のログイン必須化により公開インスタンスはほぼ全滅。セルフホストもゲストAPIトークン廃止で動作しない
- RSSHub: Twitter ルートは内部的にスクレイピングに依存しており不安定
- 商用 RSS サービス（rss.app, fetchrss.com 等）: 内部実装は同じく非公式手段に依存。突然動かなくなるリスクあり

**評価: 実用的ではない**

### 2. スクレイピングライブラリ

- **snscrape**: かつて最も人気だったが、2023年以降動作停止。メンテナンスも停滞
- **twscrape**（Python）: 認証済みアカウントのクッキーで内部 GraphQL API を叩く。複数アカウントプール機能あり。比較的活発にメンテナンスされている
  - デメリット: アカウント凍結リスクあり。X 側の API 変更で頻繁に壊れる
- **rettiwt-api**（JavaScript）: twscrape の JS 版的な位置づけ。同様のリスクあり

**評価: 動くが不安定。アカウント凍結覚悟**

### 3. ブラウザ自動化（Playwright / Puppeteer）

ヘッドレスブラウザで X にログインし、タイムラインの DOM を解析して投稿を抽出する方法。

- メリット: API 変更の影響を受けにくい（DOM 構造が変わらない限り）
- デメリット:
  - リソース消費が大きい（メモリ・CPU）
  - ログイン必須、アカウント凍結リスク
  - DOM 構造は頻繁に変更される、セレクタのメンテナンス必要
  - データセンター IP は bot 検出されやすい
  - 1ユーザー取得に数十秒かかる

**評価: 短期的には動くがメンテナンスコストが高い**

### 4. サードパーティサービス（有料）

- **Apify**（Twitter Scraper アクター）: クラウドスクレイピング基盤。メンテナンスを Apify 側が行う。API 経由で結果取得可能。$49/月〜（無料枠あり）
- **SocialData.tools**: X の非公式 API ラッパー。$9/月で10,000リクエスト程度
- **Bright Data / Oxylabs**: 大規模向け、個人利用にはオーバースペック

**評価: お金で安定性を買うパターン。GitHub Actions との相性も良好**

### 5. その他（実用性低）

- IFTTT / Zapier: 現在は公式 API 経由のため結局 API 制限に縛られる
- Google キャッシュ / Wayback Machine: リアルタイム取得には不向き

---

## 利用規約上のリスク（全手法共通）

- X の利用規約はスクレイピングを明確に禁止している
- 認証アカウントを使う手法ではアカウント凍結の可能性がある
- X はスクレイピング企業を積極的に提訴する姿勢（2023年に複数企業を提訴）
- データセンター IP は積極的にブロックされる

## 現実的な選択肢

### A. ローカル Electron アプリで半自動収集（有力候補）

自分の PC 上で Electron（または Playwright `headless: false`）を使い、自分のアカウントでログイン済みの X を自動操作してポストを取得する方式。

- 住宅 IP で実行するためデータセンター IP のようにブロックされにくい
- 本物の Chromium なので bot 検出に引っかかりにくい
- 自分のアカウントでログイン済みの状態を使うので捨てアカウント不要
- PC で手動起動する必要がある（完全自動の定期実行にはならない）が、「ボタン一発で収集」の半自動ツールとしては十分実用的
- Electron なら収集状況を表示する UI もつけやすい
- DOM 構造変更への追従は必要（どの手法でも同じ）

### B. Apify（有料）

メンテナンス不要で安定。コストはかかる（$49/月〜）。

### C. twscrape + 捨てアカウント（無料）

無料だが不安定、アカウント凍結覚悟、自前メンテナンス必要。
