# パレ学マスター 2nd Season（仮）仕様書

## 1. プロジェクト概要

### 1.1 プロジェクト名
パレ学マスター 2nd Season（仮）

### 1.2 コンセプト

パレデミア学園のタレントを「知る・覚える・好きになる」ためのWebクイズゲームの続編。

> **用語の統一**
> - ゲームを遊ぶ人 → **プレイヤー**
> - パレデミア学園所属VTuber → **タレント**

#### 目的（優先順位順）

1. **2期生の顔と名前を楽しく覚えてもらう**（最重要）
   - 2nd Seasonの主役は2期生
   - 新たなパレ学ファンにも、既存ファンにも2期生を知ってもらう入口となる
2. **新たなパレ学ファンに1期生の顔と名前も覚えてもらう**
   - 2期生から入った新規ファンにも1期生を知ってもらう
3. **1期生のことをもっと深く知ってもらう**
   - 前作の深堀り問題（難易度1〜5の4択クイズ）を知識クイズとして継続・拡充
4. **2期生のことをもっと深く知ってもらう**
   - パレ学・寮に関する共通問題は前作から流用し、初期リリースから提供
   - 2期生固有の深堀り問題は十分な問題数を用意できるまで保留

#### 前作との位置づけ
- 前作「パレ学マスター」は現状のまま維持し、問題追加などは行わない
- 2nd Seasonは別アプリとしてリリースする
- 2nd Seasonの1期生対象は**46名**

### 1.3 ターゲットプレイヤー
- パレデミア学園のファン
- 前作・前々作のプレイ経験者
- 2期生から入った新規ファン
- タレント自身（配信でのプレイ、1期生が2期生を覚える、2期生が1期生や同期を覚える）

### 1.5 2nd Seasonの方向性・改善点

#### コンセプトから導かれる方針
- 2期生の「顔と名前を覚える」体験を最優先で磨く
- 1期生の顔名前当ても引き続き提供し、新規プレイヤーが1期生に親しめる導線を設計する
- 深堀り問題は1期生分を継続、2期生分は問題数が揃い次第追加
- 2期生の深堀りクイズは段階的リリースとする

#### 顔名前当ての進化

| 世代 | 方式 |
|------|------|
| 前々作 | 顔当て・名前当てが別々のゲームモード |
| 前作 | 「顔名前当て」として顔当て・名前当てがランダムに混合出題 |
| **2nd** | 顔当て・名前当てに加え、**3〜8種の新しい出題パターン**を追加し、ランダムに混合出題 |

- 各問題タイプごとに3段階の難易度を設ける（従来通り）
- 問題タイプの追加が容易なプラグイン的アーキテクチャを採用する
  - 共通インターフェースを定義し、個別の問題タイプの開発を独立して行えるようにする
- 問題タイプの一覧は §2.1 で定義
- 設定画面で問題タイプの選択（チェックボックスでON/OFF）ができるようにする
  - 初期状態では従来の顔名前当てのみ有効
  - 特定条件（アチーブメント達成等）を満たすと他の問題タイプが解放される

#### タイムアタックモード（隠し要素）
- アチーブメント等の条件で解放される隠しモード
- 不正解時にペナルティとして秒数加算
- タイムアタック専用の出題パターンあり

#### アチーブメント
- 前作よりシンプルな形にする
- 前作はアチーブメント用画像の用意が大変だったため、画像に頼らない設計にする

#### UI/UXの方向性
- **全体的に明るく、春っぽく、ポップなデザイン**
- デザイン生成は Gemini の Nano Banana Pro を活用予定

#### 応援演出
- 2期生の問題が出題されているとき、1期生がランダムに画面の端で応援してくれる
- 1期生の問題が出題されているとき、2期生がランダムに画面の端で応援してくれる

#### ストーリー・読み物
- **1期生お迎えショートストーリー**: 1期生が2期生を迎え入れるような物語（案）
- **開発日誌**: AIが執筆する
- **スタッフロール**: 作成する。今回の主役は Claude Code × Claude Opus 4.6

#### その他
- **タレント一覧**: 作成する
- **ヘルプ**: 作成する
- **ゲーム終了画面**: いい感じにする（詳細は §4 で検討）
- **SEO**: 前作・前々作ともにSEOがうまくいっていないため、今作では対策を講じる

## 2. 機能要件

### 2.1 コア機能

| # | 機能名 | 概要 | 前作からの変更 | 優先度 |
|---|--------|------|---------------|--------|
| 1 | 顔名前当て（問題タイプ混合） | 下記問題タイプがランダムに出題される | 継続 / 大幅拡張 | 最高（MVP） |
| 2 | 知識クイズ | 4択の知識問題。難易度★☆☆〜★★★ | 前作の基本問題+深堀りを統合 | 最高（MVP） |
| 3 | タイムアタック | 隠し要素。1期生+2期生混合の最終試練 | 新規 | 中（ポストリリース） |

#### タイムアタックモード

##### 導線・解放
- タイトル画面に独立ボタンとして配置（解放後に出現）
- 解放条件: 全バッジスロットをゴールドにする
- 通常モードでは排除した1期生+2期生混合が、タイムアタック限定で登場する

##### ゲーム仕様
- **問題数**: 固定100問
- **出題構成**: 全問題タイプ（顔名前当て系すべて + タイムアタック専用 + 知識クイズ）を決まった順番で決まった数ずつ出題。レベルを段階的に上げていく
- **難易度**: 固定（100問の中にLv.1〜Lv.3が含まれる）
- **制限時間**: なし（全100問クリアまでの合計タイムを計測）
- **不正解ペナルティ**: タイムに秒数加算（例: +5秒）
- **回答後の進行**: 回答後タイマー一時停止 → 解説確認可能 →「次へ」ボタンでタイマー再開
  - 配信者が問題について話す時間を確保できる
  - RTA的に遊びたい人は即座に「次へ」を押せばよい

##### 出題構成イメージ（問題タイプ数や配分は確定後に調整）
```
Lv.1 (★☆☆相当): 各タイプ × 数問 = 約33問
Lv.2 (★★☆相当): 各タイプ × 数問 = 約33問
Lv.3 (★★★相当): 各タイプ × 数問 = 約34問
合計: 100問
```

##### ランキング・記録
- 自己ベストタイムをLocalStorageに保存
- Xに結果を投稿する導線を用意（ハッシュタグで擬似ランキング）
- 余裕があればGoogle Sheets連携等で簡易ランキングを検討

#### 知識クイズの統合方針
- 前作の「基本問題」（difficulty 0）と「深堀り問題」（difficulty 1〜5）を難易度で一本化
  - ★☆☆: difficulty 0（基本問題相当）
  - ★★☆: difficulty 1〜2
  - ★★★: difficulty 3〜5
- 2期生の知識クイズについて:
  - 2期生固有の深堀り問題は十分な問題数が用意できるまで保留
  - ただし、パレ学や寮自体に関する問題（タレント個人に依存しない問題）は前作からそのまま流用可能
  - これにより2期生の知識クイズも初期リリースからある程度の問題数で提供できる

#### 問題タイプ一覧

| # | 問題タイプ名（仮） | 概要 | 難易度変化の例 | 備考 |
|---|-----------------|------|--------------|------|
| 1 | 従来の顔名前当て | 顔→名前、名前→顔の4択 | 似た髪色の選択肢 / シルエット化 | 前作からの継続 |
| 2 | 名前の文字を集める | タレント名の文字+ダミー文字がバラバラに配置。順番にクリックして名前を完成させる。回答ボタンで判定：正解文字は固定、位置違いは色付きで戻る、不正解は消滅 | ダミー文字の増加 | ワードルに近い判定ロジック |
| 3 | タイピング | タレントの名前をタイピングで入力 | 読みがな非表示 / 名前非表示（画像のみ） / シルエット化 | PC向け。スマホでの体験は要検討 |
| 4 | タレント紹介クイズ | 先輩タレントがプロフィールを紹介→微妙に違う名前3つを含む4択、またはプロフィールに関する問題 | 紹介内容の情報量を減らす等 | 「覚える→試す」の学習サイクル |
| 5 | 名前はどこ？ | 文字グリッドの中から指定タレントの名前を見つけてクリック | グリッドサイズの拡大 / ダミー名の増加 | 利用規約の同意確認風 |
| 6 | スポットライト | タレント画像のスポットライトが当たった部分だけ見える状態で名前を当てる | スポットライトの範囲縮小 | |
| 7 | モザイク/ぼかし/ズームアップ | 加工された画像がだんだん正常になっていく過程で名前を当てる | 初期状態の加工度合い | 早く答えるほど高評価等の要素も可 |
| 8 | スクラッチ | 画像をスクラッチで削って少しずつ見えるようにし、名前を当てる。スクラッチ幅は狭め | スクラッチ幅の調整 | プレイヤーの能動的操作が楽しい |

※ 上記すべてを初期リリースに含めるかは優先度を見て判断する
※ タイムアタック専用の問題タイプをこの中から選定、または別途追加する
※ 初期状態では問題タイプ1（従来の顔名前当て）のみ有効。他は特定条件で解放

### 2.4 難易度・問題数の設計

#### 顔名前当て
- 難易度3段階（★☆☆ / ★★☆ / ★★★）
- 問題数の選択肢は不要（出題範囲の全タレントが出題される）

#### 知識クイズ
- 問題数: 10問 / 30問 / 50問 から選択
  - 問題プールが選択肢の数を下回る場合はその選択肢を非表示
- 難易度は1期生/2期生で異なる:
  - **1期生**: 3段階（★☆☆=difficulty 0 / ★★☆=difficulty 1-2 / ★★★=difficulty 3-5）
  - **2期生**: 難易度選択なし（問題数が少ないため1段階）
    - 2期生固有の深堀り問題が増えた段階で難易度を追加

### 2.7 ゲーミフィケーション要素

#### アチーブメント概要
- 前作（27個、個別画像必要）よりシンプルな構成
- 画像は少数（10枚以内）で、進化段階はCSSエフェクト（光沢・キラキラ・サイズ変化）で表現
- 前々作のように同じバッジが進化していく方式

#### バッジの種類と画像

| # | 画像 | 用途 | 枚数 |
|---|------|------|------|
| 1 | クリアバッジ | 顔名前当て用（10スロットで共有） | 1枚 |
| 2 | 知識バッジ | 知識クイズ用（2スロットで共有） | 1枚 |
| 3 | 2期生マスター称号 | 複合アチーブメント | 1枚 |
| 4 | 1期生マスター称号 | 複合アチーブメント | 1枚 |
| 5 | パレ学マスター称号 | 最終複合アチーブメント | 1枚 |
| 6 | タイムアタック解放記念 | 隠し要素解放 | 1枚 |
| 7-10 | 予備 | 問題タイプ全解放記念、寮紋章、タイムアタックランク等 | 4枚 |

#### バッジの進化段階

### 2.8 ストーリー・演出要素

#### 基本方針
- ゲーム全体を貫くストーリーや、タレント個人に台詞を喋らせるような演出は行わない
  - タレントから苦情が来るリスクを避ける
- その分、クリア演出を豪華にする方向で差別化する

#### クリア演出（全問正解時）
- 特別な演出（紙吹雪、バッジ進化アニメーション等）
- 応援タレントが祝福してくれる演出
- スキップ可能

#### 応援演出
- 2期生の問題出題中に1期生がランダムに画面端で応援
- 1期生の問題出題中に2期生がランダムに画面端で応援
- 台詞は使わず、立ち絵のみ（またはシンプルなリアクション）

#### 開発日誌
- AI（Claude Opus 4.6）が丁寧語で執筆する実際の開発記録
- 最初から全エントリ閲覧可能（隠し要素にしない）
- リアルタイムの日付で、実際の開発過程を記録
- 日付は暦日ではなく「寝るまで」を1日とする（深夜0時を過ぎても作業継続中なら前日の日付）

#### AIの手記（仮称・検討中）

プレイヤーの行動が反映されるパーソナライズされたハートフルストーリー。
ゲーム進行のマイルストーンごとに1エントリずつ解放され、最終的に一本の物語として完成する。

**基本コンセプト:**
- AI（私）がプレイヤーのプレイを見守り、感じたことを手記として綴る
- テンプレート＋実データの差し込みにより、プレイヤーごとに内容が異なる
- 口調は丁寧語。開発日誌と同じAIが書いている設定
- 一度生成されたエントリは固定（再プレイで上書きしない）

**エントリ例（イメージ）:**
```
第1話（初回プレイ完了時）
「今日、初めてのプレイヤーが来てくれました。最初に挑戦したのは
 ◯◯寮の顔名前当て。初めて正解した2期生は◯◯◯◯◯でした。
 この名前は、きっとずっと覚えていてくれるのではないでしょうか。」

第N話（タイムアタック初挑戦時）
「ついにタイムアタックに挑んでもらえました。記録は◯分◯秒。
 106人全員と向き合った◯◯分間は……」

最終話（パレ学マスター達成時）
「106人全員の顔と名前を覚えた人がここにいます。……」
```

**実装方針:**
- マイルストーン達成時のデータ（タレント名、寮、記録等）をLocalStorageに保存
- エントリ数・トリガー条件はアチーブメント設計確定後に決定
- ストーリーの詳細な内容は開発後半で執筆

#### その他の隠し要素
- スタッフロール: パレ学マスター称号で解放

### 2.9 進捗の永続化
- バッジ獲得状況: LocalStorage
- 問題タイプ解放状況: LocalStorage

```jsonc
      "eyeColorRight": "blue",    // 右目の色（オッドアイ対応）
      "mbti": "INFJ",             // MBTI（空文字列可）

      // === ファン情報 ===
      "fanName": "ひかりっ子",     // ファン呼称（空文字列可）
      "fanMark": "⭐",            // ファンマーク（空文字列可）

      // === 趣味・特技等 ===
      "hashtags": [               // ハッシュタグ（用途付き）
        { "tag": "#星空ひかり", "usage": "" },
        { "tag": "#ひかりの絵", "usage": "ファンアート" },
        { "tag": "#ひかりの配信", "usage": "配信" }
      ],
      "hobbies": ["歌", "星を見ること"],
      "skills": ["即興ソング"],
      "favorites": ["プリン", "星のカービィ"],

      // === SNS・リンク ===
      "links": [
        { "type": "official", "url": "https://..." },
        { "type": "youtube",  "url": "https://..." },
        { "type": "x",        "url": "https://..." },
        { "type": "tiktok",   "url": "https://..." },
        { "type": "marshmallow", "url": "https://..." }
      ]
      // ※ 新しいSNSが追加されてもスキーマ変更不要
    }
  ]
}
```

**前作からの主な変更点:**

| 項目 | 前作 | 2nd Season | 理由 |
|------|------|-----------|------|
| ID | 暗黙（配列index） | `id`（学籍番号） | 安定した参照 |
| 世代 | なし | `generation` | 1期生/2期生の区別 |
| 姓名 | `name_separated`（文字列） | `familyName` + `givenName` | name_separatedをスペースで分割。洋風名は正確でない場合がある |
| 寮 | 正式名のみ | コード（`"co"`等） | questions.jsonとの統一 |
| 目の色 | なし | `eyeColorLeft` + `eyeColorRight` | オッドアイ対応 |
| 血液型 | なし | `bloodType` | プロフィール拡充 |
| ハッシュタグ | 文字列配列 | `{tag, usage}` 配列 | 用途別分類対応 |
| SNS | 個別フィールド×5 | `links` 配列 | SNS追加時にスキーマ変更不要 |
| awards | 文字列配列 | 別ファイルに分離 | 構造化・共有データとして管理 |
| キー命名 | snake_case混在 | camelCase統一 | TypeScript親和性 |

#### 寮マスタ

寮コードと表示名の対応はソースコード内の定数で定義する。

```typescript
export const DORMITORIES = {
  co: { name: 'クゥ寮', order: 0 },
  me: { name: 'ミュゥ寮', order: 1 },
  wa: { name: 'バゥ寮', order: 2 },
  wh: { name: 'ウィニー寮', order: 3 },
} as const
```

2期生の寮構成が1期生と異なる場合は、世代ごとに寮マスタを持てるようにする。寮数・寮名をハードコードしない。

### 6.3 問題データ（questions.json）

知識クイズの問題データ。顔名前当ての問題は talents.json から動的に生成されるため、ここには含まない（前作踏襲）。

```jsonc
{
  "version": 2,
  "questions": [
    {
      "id": "q001",              // 一意ID
      "generation": 1,           // 対象世代（1=1期生, 2=2期生, 0=世代無関係）
      "question": "この中で身長が一番高いのは？",
      "answers": [               // 選択肢（[0]が正解、表示時にシャッフル。前作踏襲）
        "桜雲ほわり",
        "月読みるく",
        "猫乃間うと",
        "姫宮ねね"
      ],
      "difficulty": 2,           // 難易度（0〜8。前作は0〜5だったが拡張可能に）
      "genre": "プロフィール",    // ジャンル（表示用。判定には使わない）
      "sortAnswers": false,      // 選択肢をソートして表示するか
      "hideIcon": false,         // タレントアイコンを回答前に隠すか
      "image": null,             // 問題画像ファイル名（null=なし）
      "answerPool": "",           // 空の選択肢を自動補完するプール名（後述）
      "comment": "桜雲ほわりは172cmで最も背が高い",
      "sourceUrl": ""            // 出典URL
    }
  ],
  "answerSets": {               // カスタム選択肢セット（寮・世代で表現できないグループ）
    "声劇サークル": ["タレント名1", "タレント名2", "..."],
    "バレンタイングッズ化": ["タレント名1", "タレント名2", "..."]
  }
}
```

**前作からの主な変更点:**

| 項目 | 前作 | 2nd Season | 理由 |
|------|------|-----------|------|
| ID | なし | `id` | 追加・削除・並べ替えに強い |
| 世代 | なし | `generation` | 1期生/2期生+共通問題の区別 |
| category | `"基本問題"` / `"深堀り問題"` | 廃止 | difficultyで区別可能（§2.4参照） |
| difficulty | 0〜5 | 0〜8（拡張可能） | 将来の難易度追加に備える |
| dormitory | 寮コード | 廃止 | 知識クイズは寮ごとの出題がないため不要 |
| questioner | 全問空 | 廃止 | 未使用のため削除 |
| image | question直下 + comment内 `[image:]` 混在 | `image` に統一 | 二重管理の解消 |
| genre | 表記揺れあり | そのまま踏襲 | 表示用のみ。判定に使わないため正規化不要 |
| answer_set.json | 別ファイル | `answerSets` として questions.json に統合 | ファイル数削減 |
| キー命名 | snake_case | camelCase | TypeScript親和性 |

#### 選択肢の自動補完（answerPool）

知識クイズの選択肢は通常4つすべてスプレッドシートに記入するが、選択肢の一部を空にして `answerPool` を指定すると、ゲーム側が実行時にプールからランダムに補完する。

**スプレッドシートの記入例:**

| 問題文 | 選択肢1(正解) | 選択肢2 | 選択肢3 | 選択肢4 | 選択肢補完元 |
|--------|-------------|---------|---------|---------|------------|
| ミュゥ寮で一番背が高いのは？ | 月読みるく | | | | 2期生/ミュゥ寮 |
| 声劇サークルのリーダーは？ | 星空ひかり | | | | 声劇サークル |
| この中で身長が一番高いのは？ | 桜雲ほわり | 月読みるく | 猫乃間うと | 姫宮ねね | |

**answerPool の命名規則:**

| 記法 | 意味 | プールの実体 |
|------|------|------------|
| （空） | 自動補完なし | 選択肢4つすべて記入必須 |
| `2期生/ミュゥ寮` | 2期生のミュゥ寮 | talents.json から generation=2, dormitory=me のタレント名 |
| `2期生/全員` | 2期生全員 | talents.json から generation=2 のタレント名 |
| `1期生/バゥ寮` | 1期生のバゥ寮 | talents.json から generation=1, dormitory=wa のタレント名 |
| `1期生/全員` | 1期生全員 | talents.json から generation=1 のタレント名 |
| `全員` | 全タレント | talents.json の全タレント名 |
| `声劇サークル` | カスタムセット | questions.json の `answerSets` から参照 |

**ゲーム側の補完ロジック:**
1. `answerPool` が空 → 補完なし（answers をそのまま使用）
2. `answerPool` が `N期生/寮名` or `N期生/全員` or `全員` → talents.json からフィルタしてプール生成
3. それ以外 → `answerSets` からカスタムセットを参照
4. answers の空文字列をプールからランダムに選んで埋める（正解と重複しないように）
5. 補完後にシャッフルして表示

### 6.4 イベント実績データ（awards.json）

前作では talents.json 内の `awards[]` に文字列で格納していたが、構造化して別ファイルに分離する。

```jsonc
{
  "version": 1,
  "awards": [
    {
      "id": "a001",
      "eventDate": "2025-12",        // イベント開催年月（YYYY-MM）
      "eventName": "歌うま選手権",     // イベント名
      "result": "優勝",               // 順位・結果
      "talentId": "25CO001"           // 対象タレントのID（学籍番号）
    },
    {
      "id": "a002",
      "eventDate": "2026-02",
      "eventName": "バレンタイングッズ化投票",
      "result": "グッズ化決定",
      "talentId": "25WH005"
    }
  ]
}
```

**設計意図:**
- 同一イベントに複数タレントが関係する場合も個別レコードで管理
- `talentId` で talents.json と結合
- `eventDate` でソート・フィルタ可能
- タレント詳細画面で `talentId` でフィルタして表示

### 6.5 永続化データ（LocalStorage）

| キー | 内容 | 型 |
|------|------|-----|
| `parermaster2_badges` | バッジ獲得状況 | `Record<slotId, 'bronze' \| 'silver' \| 'gold'>` |
| `parermaster2_unlocked_types` | 解放済み問題タイプ | `string[]` |
| `parermaster2_settings` | 前回のゲーム設定 | `{ generation, game, range, difficulty, ... }` |
| `parermaster2_player_name` | プレイヤー名 | `string`（デフォルト: `"リスナー"`） |
| `parermaster2_time_attack_best` | タイムアタック自己ベスト | `number \| null`（ミリ秒） |
| `parermaster2_memoir` | AIの手記データ（生成済みエントリ） | `MemoirEntry[]` |

## 7. スケジュール

### 7.1 リリース方針

- **2期生お披露目**: 2026年4月中旬と予想（前年の1期生入寮式が4月18日）
- **リリース目標**: お披露目当日〜3日以内
- **戦略**: お披露目前に2期生データ以外のすべてを完成させ、お披露目後はデータ流し込みのみで即リリース

### 7.2 マイルストーン

| フェーズ | 内容 | 目標時期 | 備考 |
|----------|------|----------|------|
| Phase 1 | 仕様策定・設計 | 〜3月中旬 | 仕様書完成、デザイン発注 |
| Phase 2 | コア開発 | 3月下旬〜4月上旬 | 1期生データで全機能動作 |
| Phase 3 | 磨き込み・テスト | 4月上旬〜お披露目前日 | 1期生データで完全動作確認 |
| Phase 4 | 2期生データ投入 | お披露目当日〜+3日 | データ流し込み→テスト→リリース |
| Phase 5 | ポストリリース | リリース後 | 追加問題タイプ・隠し要素 |

### 7.3 Phase 2〜3 で完成させるもの（お披露目前）

1期生データのみで全画面・全機能が動作する状態にする。2期生のスロットはプレースホルダーで確保。

| 項目 | 内容 |
|------|------|
| タイトル画面 | 完成（2期生ボタンは「Coming Soon」表示でもテスト可能に） |
| 設定画面 | 完成（全パターンの動的切替） |
| クイズ画面 | 問題タイプ1（従来の顔名前当て）+ 知識クイズ |
| 結果画面 | 完成（Xシェア含む） |
| アチーブメント | 12スロット＋進化システム完成 |
| タレント一覧・詳細 | 1期生分完成、2期生タブは空で待機 |
| ヘルプ | 完成 |
| 談話室 | 前作踏襲で完成 |
| 応援演出 | 1期生の立ち絵で仕組み完成（2期生データ入れ替えのみで動く状態） |
| クリア演出 | 完成 |
| 問題タイプ解放 | 解放ロジック完成（解放される問題タイプが未実装でもエラーにならない設計） |
| 2期生データ受け入れ口 | talents.json・画像の差し替えだけで2期生が動く構造 |

### 7.4 Phase 4: 2期生データ投入スプリント（お披露目後）

お披露目で判明する情報を最速で収集・投入する。

| 作業 | 所要時間（目安） | 依存 |
|------|---------------|------|
| 2期生60名の名前・寮・画像の収集 | 数時間 | 公式発表 |
| talents.json への投入 | 1〜2時間 | 上記 |
| タレント画像の加工・配置 | 2〜4時間 | 上記 |
| 2期生知識クイズ問題の作成（共通問題は流用済） | 並行作業 | 寮構成確定 |
| 2期生全体の動作テスト | 1〜2時間 | 上記すべて |
| デプロイ | 数分 | 上記 |

**当日リリースのための前提条件:**
- Phase 3終了時点で1期生側が完全に動作している
- 2期生データの投入手順がドキュメント化・テスト済みである
- 画像加工のパイプライン（リサイズ・WebP変換等）が自動化されている

### 7.5 MVP範囲（初期リリース）とポストリリース

#### MVP（初期リリースに含めるもの）

| 機能 | 理由 |
|------|------|
| 問題タイプ1: 従来の顔名前当て | 最重要。これだけで「2期生の顔と名前を覚える」目的を達成できる |
| 知識クイズ | 1期生は前作データ流用、2期生は共通問題で初期提供可能 |
| アチーブメント（12スロット＋進化） | ゲーミフィケーションの柱 |
| タレント一覧・詳細 | 学習の補助 |
| 設定画面・結果画面 | 基本機能 |
| Xシェア | 拡散の柱 |
| 応援演出 | 1期生↔2期生の交流感。立ち絵のみなので低コスト |
| ヘルプ | 初回プレイヤー向け |

#### ポストリリース（段階的に追加）

| 機能 | 優先度 | 理由 |
|------|--------|------|
| 問題タイプ2〜8（文字集め、タイピング等） | 高 | リリース後1〜2週間で順次追加。飽き防止 |
| タイムアタック | 中 | 全スロットゴールドが条件なので、初日に到達するプレイヤーはいない |
| 開発日誌 | 中 | 隠し要素 |
| ショートストーリー | 低 | 問題作成・レビューに時間がかかる |
| スタッフロール | 低 | パレ学マスター称号が条件。最後に追加で間に合う |
| 追加問題タイプ（タイムアタック専用含む） | 低 | タイムアタック解放後に価値が出る |

### 7.6 リスクと対策

| リスク | 対策 |
|--------|------|
| お披露目が予想より早い（4月上旬） | Phase 2を前倒し。問題タイプ1＋知識クイズだけでもリリース可能な設計 |
| お披露目が遅い（5月以降） | 追加問題タイプの開発に時間を使える。むしろ好都合 |
| 2期生の寮構成が4寮でない | データ構造を柔軟に。寮数をハードコードしない |
| 2期生が60名でない | 人数もハードコードしない |
| お披露目当日に情報が一部しか出ない | 判明分だけで先行リリースし、残りを追加アップデート |
