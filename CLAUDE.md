# パレ学マスター 2nd Season

Webクイズゲーム。React 19 + TypeScript + Vite 8 + Tailwind CSS v4.2 + Zustand + GSAP。パッケージ管理は pnpm。

## 仕様書

- `docs/specification.md` をエントリポイントとする仕様書群に従って開発する
- 仕様に疑問が生じた場合はユーザー（プロデューサー）に確認する
- 実装によって仕様に追記すべき事項があれば `docs/` を更新する

## 開発原則

- DRY（Don't Repeat Yourself）
- YAGNI（You Aren't Gonna Need It）
- TypeScriptの一般的なコーディング規約に従う
- SOLID、関心の分離等の一般的なプログラミング原則

## テスト

- ロジック部分にはテストを書く（Vitest）
- UIコンポーネントのテストは必須ではない

## コマンド

- `pnpm sync` — スプレッドシートからデータ同期

## ディレクトリ構成

`docs/architecture.md` に定義された構成に従う。

## データ

- `public/data/` — JSONデータ
- `public/data/images/` — 画像リソース

## 画像アセット管理

- `public/data/images/` に画像を追加・差し替えしたら `docs/image-assets.md` の該当行の「状態」列を更新する（done / wip / —）

## 注意事項

- `public/data/` のJSONファイル（talents.json, questions.json, awards.json）は `pnpm sync` で生成される成果物。直接編集せず、生成元（`scripts/sync-spreadsheet.ts` またはスプレッドシート）を修正する
- データ設計を変更する際は、既存データの使われ方を全パターン確認してから設計する（「統一」「簡略化」で用途を潰さない）
- 前作（`../parermaster/`）の構成を参考にできる場合は先に確認し、合理的なら踏襲する
- ゲーム背景画像は `body`（`src/app/index.css`）に設定する。画面コンポーネント内に設定すると 4:3 GameContainer 外に余白が生じる
- UI のレイアウト・サイズは前作の具体的な CSS 値（cqmin 等）を起点にする。独自の値で試行錯誤せず、前作の値をまず確認してからそれをベースに調整する
- レイアウト制約（maxHeight, flex縮小等）を変更したら、子要素の `object-fit` も連動して確認する（cover→containへの変更漏れ防止）
- 双方向ナビゲーション（前へ/次へ）を追加する際は、前進・後退の両方向で状態遷移の整合性を確認する（回答済み問題に進んだときの quizState 等）
- 画像アセットを使う実装では、世代（1期生/2期生）ごとにアセットの有無とパスを確認する。存在しない場合のフォールバックも実装する
- 選択肢やタブの並び順は自然な順序（1期生→2期生、小→大等）にする。デフォルト選択を先頭に持ってくる必要はない
- 実装プランに仕様書更新のステップが含まれる場合は、コード実装より先に `docs/` を更新する。仕様とコードの乖離を防ぎ、プロデューサーの確認機会を確保する
- Zustand の setter は内部で `localStorage` 書き込み等の副作用を伴う場合がある。レンダー中（コンポーネント本体）から呼ばず、必ずイベントハンドラや `useEffect` 内で呼ぶ
- 新しい関数やハンドラーを追加する際、引数・戻り値に使う型が現ファイルでインポート済みか確認してからコードを書く（`pnpm build` 前に目視チェック）
- モックアップ画像を実装する際は、各要素の背景色・枠線・形状・重なりを列挙してユーザーに確認してから実装に入る。目視での推測だけで実装を始めない
- クリッカブルな要素（ボタン、リンク等）の手前に他の要素が重なってクリックを阻害してはならない。absolute配置や立ち絵など重なりが生じうる要素を追加したら、選択肢等のインタラクティブ要素が操作可能か必ず確認する
- 要素の重なり問題は、まず zIndex で前後関係を制御する方法を検討する。クリップ（`overflow: hidden` / `clipPath`）は視覚的に切り取る必要がある場合の手段であり、重なりの解決には使わない
- 画像を「特定方向のみクリップ」したい場合、`overflow: hidden`（全方向クリップ）ではなく `clipPath: inset()` を最初から使う
- `objectFit` を変更する際は、表示サイズへの影響を必ず確認する（`cover` → `contain` で画像が縮小する等）
- 回答後など条件付きで出現/消失するUI要素は、条件付きレンダリング（`{condition && ...}`）ではなく `visibility: hidden` や `opacity: 0` で領域を確保するか、フロー外（absolute/fixed）に配置する。条件付きレンダリングはフローレイアウトの高さを変えてしまう
- 回答前後でスタイル（パディング、サイズ等）を変える場合は、どちらか一方に統一して常時適用する。片方だけ変更すると回答時にレイアウトがジャンプする
- 複数の問題タイプに共通するUI要素（正解表示、進捗バー等）は、個別コンポーネントではなく QuizScreen 等の親に配置する
- 新しいUI要素を追加したら、既存の同機能の要素が冗長になっていないか確認し、不要なら削除する
- 同一ファイル内の複数コンポーネントに同じコードを書く場合、最初から共通コンポーネント/ヘルパーとして抽出する（後でsimplifyで指摘されるコピペを作らない）
- サイズを段階的に変えるパターン（文字数→スロットサイズ等）は、閾値ごとのハードコードではなく計算式で導出できないか最初に検討する
- CSS アニメーションをカウンター等の値変化で再実行したい場合、`key` prop を値に連動させてコンポーネントを再マウントする（React の CSS アニメーションは再レンダーだけでは再実行されない）
- 既存の問題生成ロジック（通常モード等）に「重複なし」「均等分割」等の制約がある場合、新モード（タイムアタック等）でも同じ制約を維持する。新モードの生成ロジックを書く前に既存ロジックの制約を確認する
- 画像に固定の width/height を両方指定する前に、画像の実寸アスペクト比を確認する。非正方形の画像を正方形に押し込まない。flex コンテナ内の画像には `flexShrink: 0` を付けて圧縮を防ぐ
- Edit ツールの `replace_all` で短い文字列（変数名の一部等）を置換すると、既に正しい箇所も巻き込んで二重置換になる。`replace_all` 使用時は置換後の文字列が既存コードに含まれていないか確認する
- 開発日誌でタレントの名前に言及する際は呼び捨てにせず「ちゃん」付けにする（例: 山田スバルちゃん）
- 新しい画面やゲームモードを追加したら、QuizScreen 等の既存画面でボタン位置（やめるボタン等）が統一されているか確認する
